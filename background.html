Zoned Storage Devices란? — 등장 배경과 성능 이점

Zoned Storage Device는 LBA(논리 블록 주소) 공간을 Zone이라는 고정 크기 영역으로 나누고, 각 Zone 내에서는 반드시 순차적으로(sequentially) 써야 한다는 제약을 갖는 저장 장치입니다.

Device LBA range divided in zones
Zone 0
Zone 1
Zone 2
Zone 3
…
Zone X
← ZONE RESET ▲ Write Pointer WRITE →
ZONE RESET은 Write Pointer를 Zone 시작으로 되돌리고, WRITE는 Write Pointer를 앞으로 전진시킨다.

왜 Zoned Storage가 필요한가?

일반 NVMe SSD(Conventional Block Device)는 호스트가 임의의 LBA에 자유롭게 읽고 쓸 수 있는 랜덤 쓰기 인터페이스를 제공합니다. 이를 위해 내부 FTL(Flash Translation Layer)이 논리 주소와 물리 플래시 주소 간 매핑, 가비지 컬렉션(GC), 웨어 레벨링을 수행합니다. 그러나 FTL의 GC는 다음 문제를 일으킵니다.

Zoned Storage의 이점메커니즘
Housekeeping 연산 감소 순차 쓰기 제약으로 내부 GC Copy-back 제거 → WAF ≈ 1.0
Over-provisioning 감소 GC용 여유 공간 불필요 → 실제 용량 증가, 비용 절감
I/O 응답 예측성 향상 GC로 인한 갑작스러운 Tail Latency 스파이크 제거
02

예시 장치 — SMR HDD와 NVMe ZNS SSD

SMR(Shingled Magnetic Recording) HDD

자기 기록 방식에서 트랙을 기왓장(shingle)처럼 겹쳐 쌓아 기록 밀도를 높인 HDD입니다. 트랙이 겹치기 때문에 임의 위치에 쓰면 인접 트랙 데이터가 손상됩니다. 따라서 Zone 단위로 순차 쓰기만 허용하며, Zone 전체를 지운 후 다시 쓰는 패턴을 강제합니다. Zoned Storage 개념이 처음으로 HDD에 적용된 사례로, 표준은 ZBC(Zoned Block Commands)입니다.

NVMe ZNS(Zoned Namespace) SSD

NVMe 표준에 정의된 ZNS Command Set을 지원하는 SSD입니다. SMR HDD와 같은 Zone 제약을 NVMe 인터페이스로 노출하여 낸드 플래시 특성에 맞는 최적 I/O 패턴을 호스트에 강제합니다. 현재 데이터센터용 제품(예: Samsung ZNS SSD, WD Ultrastar DC ZN540)이 출시되어 있습니다.

ℹ️ 본 실습에서는 실제 ZNS SSD 대신 null_blk 커널 모듈로 ZNS 인터페이스를 에뮬레이션합니다(/dev/nullb0). 인터페이스 동작은 동일하나, 메모리 기반이므로 실제 낸드 지연 시간은 반영되지 않습니다.
🔬 Lab 연결: blkzone report /dev/nullb0 출력에서 type: 2(SEQ_WRITE_REQUIRED)를 확인하면 이 장치가 ZNS Zone임을 알 수 있습니다. Lab 2 Step 2.1에서 직접 확인합니다.

ZNS SSD 핵심: 데이터 배치 제어권의 이동

ZNS SSD와 일반 SSD의 가장 근본적인 차이는 데이터가 플래시에 실제로 기록되는 위치를 누가 결정하느냐에 있습니다.

일반 SSD (Regular SSD)
App 1 · App 2 · App 3
↓ ↓ ↓
Conventional SSD Controller
Device controls data placement
ZNS SSD
App 1 · App 2 · App 3
↓ ↓ ↓
ZNS SSD Controller
Applications control data placement in zones
일반 SSD는 FTL이 내부적으로 데이터를 배치하지만, ZNS SSD는 각 애플리케이션이 자신의 Zone을 직접 관리한다.

일반 SSD에서는 FTL이 데이터 배치를 모두 결정하므로 애플리케이션 간 GC 간섭이 불가피합니다. ZNS SSD에서는 각 애플리케이션이 자신만의 Zone에 데이터를 직접 배치합니다. 서로 다른 애플리케이션의 데이터가 같은 플래시 블록에 섞이지 않으므로 GC Copy-back 없이 Zone 단위 Reset만으로 공간을 재활용할 수 있습니다.

✅ 이 특성이 WAF ≈ 1.0을 가능하게 하는 근본 원인입니다. Lab 3에서 직접 측정합니다.
04

Zone 구조 — Zone Size, Zone Capacity, Write Pointer

ZNS 장치의 LBA 공간 전체는 동일한 크기의 Zone들로 나뉩니다. 실습 환경에서는 Zone 크기 256 MB / 전체 장치 크기 10 GB이므로 총 40개의 Zone이 존재합니다.

Zone Size vs Zone Capacity

Zone Size는 Zone에 할당된 LBA 공간의 전체 크기이고, Zone Capacity는 Zone 내에서 실제 쓰기가 가능한 공간입니다. 플래시 매체의 물리적 제약으로 Zone Capacity ≤ Zone Size일 수 있으나, null_blk 에뮬레이션에서는 동일합니다. blkzone report 출력에서 len(Zone Size)과 cap(Zone Capacity)으로 확인합니다.

Write Pointer (WP)

각 Zone은 Write Pointer(WP)를 가집니다. WP는 해당 Zone에서 다음에 쓸 수 있는 LBA를 가리킵니다. 쓰기가 성공하면 WP는 쓴 블록 수만큼 자동으로 전진합니다. WP가 가리키지 않는 위치에 쓰기를 시도하면 커널 블록 레이어가 이를 거부하고 -EINVAL 오류를 반환합니다.

Zone 0
FULL
Zone 1
OPEN
Zone 2
EMPTY
Zone 0은 FULL (WP = Zone 끝), Zone 1은 OPEN (WP가 중간), Zone 2는 EMPTY (WP = Zone 시작)

Zone 관리 명령 — Reset / Open / Close / Finish

호스트는 아래 4가지 Zone 관리 명령으로 Zone의 상태를 명시적으로 제어합니다. Linux에서는 blkzone 유틸리티로 이 명령들을 발행합니다.

RESET ZONE
Write Pointer를 Zone의 시작 위치로 이동합니다.
Zone 내의 모든 데이터가 삭제됩니다.
상태: Any → EMPTY
blkzone reset [--offset --length] DEV
OPEN ZONE
앱이 해당 Zone에 쓰기 연산을 수행할 것임을 장치에 명시적으로 알립니다.
장치는 이 Zone에 내부 자원을 미리 할당합니다.
상태: EMPTY / CLOSED → EXPLICIT OPEN
blkzone open --offset --length DEV
CLOSE ZONE
다음 OPEN 명령 전까지 이 Zone에 쓰기를 수행하지 않겠다고 선언합니다.
Write Pointer 위치는 유지되며 내부 자원이 해제됩니다.
상태: OPEN → CLOSED
blkzone close --offset --length DEV
FINISH ZONE
Write Pointer를 Zone의 끝으로 이동합니다.
Zone을 완전히 봉인(seal)하여 Reset 전까지 추가 쓰기를 차단합니다.
상태: Any → FULL
blkzone finish --offset --length DEV

07

Sequential Write 제약과 동작 방식

ZNS의 Sequential Write Required (SWR) Zone에 쓰기를 할 때, 모든 쓰기는 반드시 현재 WP가 가리키는 위치에서 시작해야 합니다. 이를 Sequential Write Constraint라고 합니다.

Sequential Write 제약의 부작용 — 낮은 병렬성

Sequential Write에서 호스트는 Zone당 WP를 추적하고 보장해야 합니다. 특히 한 Zone에 발행 가능한 동시 I/O는 사실상 1개로 제한됩니다. (WP 순서를 보장하려면 이전 I/O가 완료된 후 다음 I/O를 발행해야 하기 때문) 이로 인해 소형 블록 쓰기에서는 성능이 저하될 수 있습니다.

Sequential Write — 정상 흐름 vs 위반 예시
Zone 0, WP = 0x0000
  쓰기 I/O: LBA 0x0000, len 64KB → 성공, WP → 0x0020
  쓰기 I/O: LBA 0x0020, len 64KB → 성공, WP → 0x0040
  쓰기 I/O: LBA 0x0060, len 64KB → 오류! (WP=0x0040, 0x0060에는 쓸 수 없음)
                                     커널: -EINVAL (Invalid argument)

fio의 zonemode=zbd

fio에서 zonemode=zbd 옵션을 사용하면 fio 내부적으로 Zone당 하나의 I/O만 순서에 맞게 발행하도록 관리합니다. numjobs=1 환경에서는 이 덕분에 QD를 높여도 Sequential Write 오류 없이 처리량을 향상시킬 수 있습니다.

⚠️ zonemode=none으로 ZNS 장치에 임의 쓰기를 시도하면 커널 블록 레이어가 WP 위반을 감지하여 -EINVAL 에러를 반환합니다. Lab 1 Step 1.2에서 이 에러를 의도적으로 유발합니다.
🔬 Lab 연결: Lab 1의 모든 fio 명령에 --zonemode=zbd가 포함됩니다. Step 1.2에서 zonemode=none randwrite로 -EINVAL 에러를 직접 확인합니다. (Lab 1 분석 Q3: 이 상황에서 dm-zoned나 F2FS가 어떤 역할을 하는지 → 섹션 10 참조)

Zone Append — 병렬 쓰기와 원자성

Sequential Write의 "Zone당 동시 I/O = 1" 제약을 극복하기 위해 NVMe ZNS Command Set은 Zone Append 명령을 정의합니다. 호스트는 정확한 LBA를 지정하지 않고 "이 Zone에 붙여서 써줘"라고 요청합니다. SSD 컨트롤러가 WP를 원자적으로 갱신하고, 실제 기록 위치를 응답으로 반환합니다.

Zone Append에서는 B, C의 순서가 A와 달리 결정되었으나 WP는 정확하게 전진. 호스트는 응답 LBA를 추적해야 함.

핵심 특성 1: 원자성 (Atomicity)

WP 갱신이 장치 내에서 원자적으로 수행됩니다. 여러 호스트 I/O가 동시에 발행되더라도 장치가 순서를 결정하고 겹치지 않는 LBA를 각각 할당합니다. 호스트 측에서 순서를 직접 제어할 필요가 없습니다.

핵심 특성 2: LBA 반환 — 호스트의 추적 의무

데이터가 실제로 기록된 위치를 호스트가 모를 수 있으므로, Zone Append 명령의 응답에는 기록된 LBA가 포함됩니다. 파일 시스템(예: F2FS)이나 애플리케이션은 이 반환 LBA를 저장해두어야 이후 Read가 가능합니다.

Append Size Limit (ZASL)

한 번의 Zone Append 명령으로 쓸 수 있는 최대 크기를 ZASL(Zone Append Size Limit)이라 합니다. 장치마다 다르며, nvme id-ns /dev/nullb0 --human-readable에서 zasl 필드로 확인합니다.

09

Active / Open Zone 제한

ZNS 장치는 동시에 열어 둘 수 있는 Zone의 수를 제한합니다. 이는 장치 내부 Write Buffer 등 자원 때문입니다.

파라미터NVMe 명세 이름의미실습 환경 값
zone_max_open MOR (Maximum Open Resources) 동시에 OPEN 상태일 수 있는 최대 Zone 수 14
zone_max_active MAR (Maximum Active Resources) 동시에 OPEN 또는 CLOSED(Active)일 수 있는 최대 Zone 수 14

이 제한을 초과하면 새 Zone을 열기 전에 기존 Zone을 CLOSE 또는 FINISH 해야 합니다. fio의 zonemode=zbd는 이 제한을 자동으로 준수합니다. nvme id-ns 출력에서 mor, mar 필드로 확인할 수 있습니다.

성능 측정 지표 — Bandwidth, Latency, Queue Depth

Bandwidth (처리량, MB/s)

단위 시간당 전송된 데이터 양입니다. fio 출력에서 write: bw=XXX MiB/s로 확인합니다. 일반적으로 QD가 높을수록, 장치가 병렬 처리를 지원할수록 Bandwidth가 증가합니다.

Latency (지연 시간, µs)

하나의 I/O 요청이 발행된 후 완료될 때까지의 시간입니다. fio 출력에서 lat (usec): min=..., max=..., avg=...로 확인합니다. QD가 높아지면 큐 대기 시간이 늘어나 평균 Latency도 증가합니다. ZNS는 GC가 없으므로 Conventional SSD 대비 Tail Latency(최대 지연)가 크게 낮습니다.

Queue Depth (QD)와 Little's Law

Queue Depth는 장치에 동시에 발행되는 I/O 요청의 수입니다. QD와 Bandwidth·Latency의 관계는 Little's Law로 설명합니다.

L = λ × W
L: 시스템 내 평균 요청 수 (≈ Queue Depth)  |  λ: 처리율 (IOPS)  |  W: 평균 응답 시간 (Latency)
→ QD ↑ = IOPS ↑ × Latency ↑  |  Bandwidth = IOPS × Block Size
fio 출력 읽는 방법
write: IOPS=1234, BW=77.1MiB/s (80.9MB/s)(2048MiB/26544msec)
         ↑ IOPS          ↑ Bandwidth (MB/s)
  lat (usec): min=101, max=2893, avg=812.34, stdev=93.21
                                  ↑ 평균 지연 시간 (µs) — 결과표에 기록
