1Step 4.1 — 스토리지 패러다임 변화 분석

1-1. 기존 CUFS의 한계: 수백 MB 규모의 L2P 매핑 오버헤드

스마트폰에 사용되는 Conventional UFS(CUFS)는 낸드 플래시 특성상 논리 주소(LBA)를 물리 주소로 변환하는 L2P 매핑 테이블을 SRAM에 보관해야 합니다. 스토리지 용량이 커질수록 이 테이블 크기가 선형으로 증가해 256GB 기기에서는 약 256MB의 SRAM을 L2P 테이블만으로 소진합니다. ZUFS는 순차 쓰기 제약을 통해 이를 수십 KB 수준으로 혁신적으로 줄입니다.

CUFS vs ZUFS — L2P 매핑 구조 비교
CUFS (Conventional UFS)
──────────────────────────────────────────────────
  Host LBA 0 → Physical Page 1047    ← 모든 페이지마다
  Host LBA 1 → Physical Page 5200       1:1 매핑 항목 필요
  Host LBA 2 → Physical Page  312
  ...
  ▶ 256GB 장치: L2P 테이블 약 256MB SRAM 필요

ZUFS (Zoned UFS)
──────────────────────────────────────────────────
  Zone 0 시작점 → Physical Block X
  Zone 내부는 순차 쓰기만 → 오프셋으로 위치 계산 가능

  ▶ 256GB 장치: L2P 테이블 수십 KB SRAM만 필요
    (Zone 수 × 포인터 크기 — Zone당 1항목)
💡 핵심 아이디어: ZUFS는 쓰기를 순차적으로만 허용함으로써 L2P 매핑을 Zone 단위로만 유지하면 되어 SRAM 사용량을 수십~수백 배 줄입니다.

1-2. CUFS vs ZUFS 핵심 차이

2Step 4.2 — 3대 과제 분석

ZUFS 상용 배포의 3대 장벽

개념은 간단하지만, 실제 스마트폰에 ZUFS를 탑재하려면 세 가지 비자명한(non-trivial) 문제를 해결해야 합니다.

🧠
과제 1: 제한된 SRAM 관리
순차 쓰기를 지원하려면 다수의 Zone을 동시에 열어야(Open) 합니다. Zone을 열 때마다 데이터를 임시 버퍼링할 SRAM이 필요하지만, 모바일 기기의 SRAM은 극히 제한적입니다.
🔒
과제 2: 쓰기 순서 보장
ZUFS는 순차 쓰기만 허용하므로 쓰기 순서 위반이 치명적입니다. 블록 레이어, 드라이버, 펌웨어를 거치는 동안 I/O 요청 순서가 뒤바뀌면 데이터 손상이 발생할 수 있습니다.
🗑️
과제 3: GC 오버헤드 완화
Zone 크기가 매우 크기 때문에, 유효 데이터가 조금이라도 남아 있으면 Zone을 재사용하지 못합니다. 파일시스템이 GC를 늦게 실행하면 빈 Zone 부족으로 심각한 성능 저하가 발생합니다.
🔗
Lab 실습과의 1:1 연결점 요약
  • 과제 1 ← Lab 2: Zone Open/Close 상태 전이를 직접 제어하며 SRAM 버퍼의 역할을 체감
  • 과제 2 ← Lab 1: zonemode=none 랜덤 쓰기 시 발생한 -EINVAL이 I/O 재정렬 위반의 대표 사례
  • 과제 3 ← Lab 3: F2FS 레벨 WAF 측정 — ZNS F2FS는 Zone Reset으로 WAF ≈ 1.0을 달성하지만, Zone 단위 재사용 제약이 GC 타이밍 문제로 이어짐
⚠️ 이 세 과제는 서로 연결되어 있습니다. SRAM을 아끼려고 Open Zone 수를 줄이면 GC 유연성이 떨어지고, 쓰기 순서를 지키려고 큐를 제한하면 처리량이 감소합니다. 따라서 계층 간 협력 설계가 필수입니다.

3-2. 각 최적화 기법 상세

① 동적 장치-사이드 버퍼 관리 (과제 1 해결)

SRAM을 Zone별로 정적으로 할당하면 유휴 Zone이 생길 때 낭비가 발생합니다. 동적 버퍼 공유는 현재 데이터를 수신 중인 Zone에만 SRAM을 집중 할당하고, 유휴 Zone의 버퍼는 회수해 다른 Zone에 재사용합니다.

🏦 비유: 은행 창구처럼 손님(I/O)이 있을 때만 창구(SRAM)를 열고, 손님이 없으면 다른 창구로 직원을 이동시킵니다.

② 쓰기 순서 보장 메커니즘 (과제 2 해결)

블록 레이어는 성능을 위해 요청을 재정렬(reordering)할 수 있습니다. 그러나 ZUFS에서 Zone 내 순서 위반은 즉각적인 오류를 유발합니다. Zone 경계에서 배리어(barrier) 삽입과 동일 Zone 요청의 순서 보존 큐잉으로 해결합니다.

쓰기 순서 위반 시나리오와 해결
문제: 블록 레이어 재정렬 발생
──────────────────────────────────────────────
  F2FS 발행 순서: [Zone0: 0→4KB] → [Zone0: 4→8KB] → [Zone0: 8→12KB]
  블록 레이어 재정렬: [Zone0: 8→12KB] → [Zone0: 0→4KB]  ← 순서 역전!
  ZUFS 장치: 8KB 위치에 쓰기 시도 → WP=0 → 오류(-EINVAL)

해결: Zone별 순서 보존 큐
──────────────────────────────────────────────
  Zone 0 큐: [0→4KB] → [4→8KB] → [8→12KB]  순서 보장
  Zone 1 큐: [0→4KB] → ...                  Zone 간 독립 병렬 처리
  드라이버: Zone 경계 감지 시 배리어 삽입하여 플러시

③ 선제적 GC 프레임워크 (과제 3 해결)

F2FS는 기존에 빈 공간이 부족해질 때(반응적 GC) 가비지 컬렉션을 수행했습니다. ZUFS 환경에서는 Zone 크기가 크기 때문에 반응적 GC는 너무 늦습니다. 선제적 GC는 빈 Zone 수가 임계값 아래로 떨어지기 전에 미리 GC를 실행하고, Android 프레임워크와 연동해 화면이 꺼진 유휴 시간에 집중 수행합니다.

✅ 핵심: 공간이 없어서 급히 GC하는 것이 아니라, 여유가 있을 때 미리 정리해두는 것. 사용자 체감 지연을 최소화합니다.

4-2. 단편화가 ZUFS에 유리한 이유

스마트폰을 오래 사용하면 스토리지 내부에 단편화(Fragmentation)가 심해집니다. CUFS는 단편화된 환경에서 GC가 빈번히 발생해 성능이 크게 저하되지만, ZUFS는 순차 쓰기 구조 덕분에 단편화 상황에서도 성능이 안정적입니다.
