Step 2.1 — Zone 상태 확인
1

초기 Zone 상태 확인

blkzone report로 모든 Zone의 현재 상태, 시작 LBA, 크기, Write Pointer 위치를 확인합니다.

bash — Zone 목록 조회
# 전체 Zone 목록 (처음 5개)
sudo blkzone report /dev/nullb0 | head -5

# Zone 상태별 개수 집계
sudo blkzone report /dev/nullb0 \
  | grep -oP 'zcond:\s*\d+\s*\(\K\w+' \
  | sort | uniq -c \
  | awk '{printf "  %-8s : %s zones\n", $2, $1}'

# 총 Zone 수 확인
cat /sys/block/nullb0/queue/nr_zones
예상 출력 — 초기 상태 (모두 EMPTY)
  start: 0x000000000, len 0x020000, cap 0x020000, wptr 0x000000000 reset:0, zcond:1(em)   [type: 2(SEQ_WRITE_REQUIRED)]
  start: 0x000020000, len 0x020000, cap 0x020000, wptr 0x000000000 reset:0, zcond:1(em)   [type: 2(SEQ_WRITE_REQUIRED)]
  start: 0x000040000, len 0x020000, cap 0x020000, wptr 0x000000000 reset:0, zcond:1(em)   [type: 2(SEQ_WRITE_REQUIRED)]

  em       : 256 zones   ← 모든 Zone이 EMPTY

2

Implicit Open 상태 유발 및 확인

명시적 Open 명령 없이 쓰기를 수행하면 Zone이 자동으로 Implicit Open 상태로 전이됩니다.

bash — 쓰기로 Implicit Open 유발
# Zone 0에 32MB 순차 쓰기 (Implicit Open 유발, 절반만 채워 io 상태 유지)
sudo fio \
  --name=implicit_open \
  --filename=/dev/nullb0 \
  --ioengine=libaio \
  --direct=1 \
  --zonemode=zbd \
  --rw=write \
  --bs=64k \
  --iodepth=1 \
  --size=32m \
  --numjobs=1 \
  --output-format=normal 2>&1 | grep -E 'write:|IOPS'

# Zone 상태 재확인
sudo blkzone report /dev/nullb0 | head -3
sudo blkzone report /dev/nullb0 \
  | grep -oP 'zcond:\s*\d+\s*\(\K\w+' \
  | sort | uniq -c
예상 출력 — Implicit Open 상태
  start: 0x000000000, len 0x020000, cap 0x020000, wptr 0x000010000 reset:0, zcond:2(io)  [type: 2(SEQ_WRITE_REQUIRED)]
                                                      ↑ WP 이동: 0 → 0x10000 (32MB 기록됨)
       1 io    ← Implicit Open: 1개
     255 em    ← EMPTY: 255개
ℹ️ Implicit Open은 애플리케이션이 OPEN ZONE 명령 없이 바로 쓰기를 시작할 때 장치가 자동으로 부여하는 상태입니다.
Step 2.2 — 명시적 Zone 제어
1

Explicit Open → 쓰기 → Close → Finish 통합 실행 zone_lifecycle.sh

Zone을 명시적으로 열고 데이터를 기록하면서 Write Pointer 이동을 관찰한 뒤, Close와 Finish까지 전체 상태 전이 시퀀스를 한 번에 확인합니다.

🗂️
zone_lifecycle.sh 동작 순서
  • 전체 Zone Reset — EMPTY 상태 기준선 확인
  • Zone 1 Explicit Open — blkzone open → zcond: eo 전이 확인
  • 32MB 순차 쓰기 — Write Pointer 이동 확인 (Zone 절반 기록)
  • Zone 1 Close — blkzone close → WP 유지, zcond: cl 전이
  • Zone 1 Finish — blkzone finish → WP = Zone 끝, zcond: full 전이
zone_lifecycle.sh 
#!/usr/bin/env bash
# zone_lifecycle.sh — Zone 상태 전이 (Open→Write→Close→Finish)
# 사용법: sudo bash zone_lifecycle.sh

set -euo pipefail
DEV=/dev/nullb0

show_zone() {
    printf 'Zone 1 상태: '
    sudo blkzone report "${DEV}" | sed -n '2p' \
      | grep -oP 'wptr\s+\K0x\S+|zcond:\s*\d+\s*\(\K\w+(?=\))' \
      | paste - - \
      | awk '{printf "wptr=%-14s state=%s\n", $1, $2}'
}

printf '── [1] 전체 Reset ───────────────────────────────\n'
sudo blkzone reset "${DEV}"
show_zone

printf '── [2] Explicit Open ────────────────────────────\n'
sudo blkzone open --offset 0x20000 --length 0x20000 "${DEV}"
show_zone

printf '── [3] 32MB 쓰기 ────────────────────────────────\n'
sudo fio \
  --name=write_z1 \
  --filename="${DEV}" \
  --ioengine=libaio --direct=1 --zonemode=zbd \
  --rw=write --bs=4m --iodepth=1 \
  --offset=64m --size=32m --numjobs=1 \
  --output-format=terse 2>/dev/null | cut -d';' -f6 \
  | xargs -I{} printf '  BW: {} KiB/s\n'
show_zone

printf '── [4] Close Zone ───────────────────────────────\n'
sudo blkzone close --offset 0x20000 --length 0x20000 "${DEV}"
show_zone

printf '── [5] Finish Zone ──────────────────────────────\n'
sudo blkzone finish --offset 0x20000 --length 0x20000 "${DEV}"
show_zone
printf '\n✔ 완료\n'
bash — 호스트에서 파일 전송 
scp -P 2222 zone_lifecycle.sh user@localhost:~/
✅ chmod +x zone_lifecycle.sh && sudo bash zone_lifecycle.sh
예상 출력 — zone_lifecycle.sh

  BW: 636483 KiB/s

Step 2.3 — Zone Reset (자원 회수)
1

Zone Reset 통합 실행 zone_reset.sh

단일 Zone Reset과 전체 Zone 일괄 Reset을 순서대로 실행하고, Write Pointer가 Zone 시작으로 복귀하여 EMPTY 상태로 전환됨을 확인합니다.

zone_reset.sh 
#!/usr/bin/env bash
# zone_reset.sh — 단일 Zone Reset → 전체 Zone Reset
# 사용법: sudo bash zone_reset.sh

set -euo pipefail
DEV=/dev/nullb0

# ── 1) Reset 전 Zone 1 상태 확인 ─────────────────────
printf '[1/3] Reset 전 Zone 1 상태:\n'
sudo blkzone report "${DEV}" | sed -n '2p'

# ── 2) Zone 1 단일 Reset ─────────────────────────────
printf '\n[2/3] Zone 1 Reset...\n'
sudo blkzone reset --offset 0x20000 --length 0x20000 "${DEV}"
sudo blkzone report "${DEV}" | sed -n '2p'

# ── 3) 전체 Zone 일괄 Reset ──────────────────────────
printf '\n[3/3] 전체 Zone 일괄 Reset...\n'
sudo blkzone reset "${DEV}"

sudo blkzone report "${DEV}" \
  | grep -oP 'zcond:\s*\d+\s*\(\K\w+' \
  | sort | uniq -c \
  | awk '{printf "  %-8s : %s zones\n", $2, $1}'

printf '\n처음 3개 Zone WP 확인:\n'
sudo blkzone report "${DEV}" | head -3
printf '\n✔ 모든 Zone EMPTY 복구 완료\n'
bash — 호스트에서 파일 전송 
scp -P 2222 zone_reset.sh user@localhost:~/
✅ chmod +x zone_reset.sh && sudo bash zone_reset.sh

ℹ️ Zone Reset은 데이터를 일괄 삭제하고 Write Pointer를 Zone 시작으로 되돌립니다. Conventional SSD의 Trim 명령과 유사하지만 Zone 단위로만 동작합니다.

생각해보기

Q1. Implicit Open과 Explicit Open의 차이점은 무엇이며, 언제 각각을 사용하는 것이 유리할까요?
힌트: 장치에는 동시에 열 수 있는 Zone 수(max_open_zones)에 제한이 있습니다. 명시적 제어가 이 자원 관리에 어떤 도움을 줄까요?
Q2. Zone Reset이 일반 파일 삭제와 다른 점은 무엇인가요? 낸드 플래시 관점에서 설명하세요.
힌트: 일반 파일 삭제는 메타데이터만 변경합니다. Reset은 Erase Block 수준에서 무엇을 다르게 처리할까요?
Q3. Close 상태에서 Write Pointer가 유지되는 이유는 무엇이며, 이것이 Finish와 어떻게 다른가요?
힌트: Close는 자원 절약을 위한 일시 정지, Finish는 영구 봉인입니다. 각 상태에서 다음 가능한 연산을 비교해 보세요.
