실습 3: F2FS 파일시스템 레벨 WAF 측정

nullb1 재구성 · F2FS 포맷/마운트 · GC 유발 워크로드 · NVMe vs ZNS 비교 분석

실습 목표

F2FS(Flash-Friendly File System)는 로그 구조 파일시스템으로, 파일시스템 레벨에서 자체 GC(Garbage Collection)를 수행합니다. 이 GC는 NAND 펌웨어 GC와 달리 null_blk 에뮬레이터에서도 실제로 동작하므로, 파일시스템 레벨 WAF가 1보다 클 수 있습니다.

NVMe SSD(/dev/nullb0)에 F2FS를 마운트해 랜덤 쓰기 부하를 주면, F2FS GC가 유효 데이터를 세그먼트 간에 복사(Copy-back)하여 블록 장치 실제 기록량이 앱 요청량보다 커집니다. ZNS SSD(/dev/nullb1)에서는 F2FS가 Zone 단위로 GC를 수행하므로 Copy-back이 줄어들어 WAF가 낮게 측정됩니다.

WAFfs = 블록 장치 실제 기록량 / 애플리케이션 요청 기록량
측정 근거: /sys/block/nullb*/stat field 7 (write_sectors) 전후 차이

Step 3.1 — 환경 준비
1

nullb1 재구성 및 F2FS 포맷/마운트 setup_f2fs.sh

F2FS는 ZNS 장치에서 슈퍼블록, 체크포인트 등 메타데이터를 저장하기 위해 최소 1개의 Conventional Zone을 요구합니다. 실습 1·2에서 사용한 nullb1은 Sequential Zone만 32개였으므로, zone_nr_conv=1을 추가하여 재구성해야 합니다.

변경 전 (lab1/lab2) zone[ 0~ 31] = 32개 Sequential ← 모두 순차 쓰기 전용 변경 후 (lab3) zone[ 0 ] = 1개 Conventional ← F2FS 메타데이터 저장소 zone[ 1~ 31] = 31개 Sequential ← 데이터 영역 (총 용량 동일)
⚙️
setup_f2fs.sh 동작 순서
  • nullb1 (zns0 configfs) — power=0 후 재생성, zone_nr_conv=1 추가
  • f2fs-tools 설치 여부 확인 (mkfs.f2fs -V)
  • 마운트 포인트 /mnt/conv, /mnt/zns 생성
  • nullb0 포맷: mkfs.f2fs -f /dev/nullb0
  • nullb1 포맷: mkfs.f2fs -f -m /dev/nullb1 (ZNS 모드)
  • 두 장치 마운트 후 df 확인
setup_f2fs.sh 
#!/usr/bin/env bash
# setup_f2fs.sh — nullb1 재구성 및 F2FS 환경 구성
# 사용법: sudo bash setup_f2fs.sh

set -euo pipefail

# ── 1) nullb1 재구성 (zone_nr_conv=1) ────────────────
printf '[1/5] nullb1 재구성 (zone_nr_conv=1 추가)...\n'
if [ -d /sys/kernel/config/nullb/zns0 ]; then
    echo 0 > /sys/kernel/config/nullb/zns0/power
    rmdir /sys/kernel/config/nullb/zns0
fi
mkdir /sys/kernel/config/nullb/zns0
echo 2048 > /sys/kernel/config/nullb/zns0/size
echo 1    > /sys/kernel/config/nullb/zns0/zoned
echo 64   > /sys/kernel/config/nullb/zns0/zone_size
echo 1    > /sys/kernel/config/nullb/zns0/zone_nr_conv
echo 4096 > /sys/kernel/config/nullb/zns0/blocksize
echo 1    > /sys/kernel/config/nullb/zns0/power
printf '  ✔ nullb1 재구성 완료 (conv: 1개, seq: 31개)\n\n'

# ── 2) f2fs-tools 확인 ───────────────────────────────
printf '[2/5] f2fs-tools 확인...\n'
if ! command -v mkfs.f2fs &>/dev/null; then
    apt-get install -y f2fs-tools
fi
printf '  버전: '; mkfs.f2fs -V 2>&1 | head -1
printf '\n'

# ── 3) 마운트 포인트 생성 ─────────────────────────────
printf '[3/5] 마운트 포인트 생성...\n'
mkdir -p /mnt/conv /mnt/zns
printf '  /mnt/conv, /mnt/zns 준비 완료\n\n'

# ── 4) F2FS 포맷 ──────────────────────────────────────
printf '[4/5] F2FS 포맷...\n'
mkfs.f2fs -f /dev/nullb0
printf '\n'
mkfs.f2fs -f -m /dev/nullb1   # -m: ZNS 모드 (zone_nr_conv=1 필요)
printf '  ✔ 포맷 완료\n\n'

# ── 5) 마운트 ─────────────────────────────────────────
printf '[5/5] 마운트...\n'
mount /dev/nullb0 /mnt/conv
mount -t f2fs /dev/nullb1 /mnt/zns
df -h /mnt/conv /mnt/zns
printf '\n  ✔ 마운트 완료\n'
printf '\n다음 단계: sudo bash workload.sh\n'
bash — 호스트에서 파일 전송 
scp -P 2222 setup_f2fs.sh user@localhost:~/
✅ chmod +x setup_f2fs.sh && sudo bash setup_f2fs.sh

Step 3.2 — GC 유발 워크로드
1

F2FS GC 유발 워크로드 실행 workload.sh

F2FS GC가 동작하려면 파일시스템이 충분히 채워진 상태에서 덮어쓰기(overwrite)가 일어나야 합니다. 먼저 1,600MB 파일을 순차 write로 채운 뒤, 같은 파일에 랜덤 4KB write를 60초 인가합니다. 덮어쓰인 페이지는 F2FS 세그먼트 내에서 무효(invalid) 상태로 남아 GC Copy-back을 유발합니다.

두 장치 모두 동일한 워크로드를 사용해 WAF 비교의 공정성을 유지합니다.

🗂️
workload.sh 동작 순서
  • nullb0/nullb1 write_sectors 초기값 기록 (stat field 7)
  • 각 장치에 1,600MB 파일 순차 fill (디스크 ~80% 점유)
  • 동일 파일에 randwrite 60초 — F2FS invalid 페이지 누적
  • sync로 writeback 플러시 후 write_sectors 최종값 기록
  • fio JSON 출력에서 앱 기록량 추출 → /tmp/*_waf_stat 저장
workload.sh 
#!/usr/bin/env bash
# workload.sh — F2FS WAF 측정 워크로드
# 사용법: sudo bash workload.sh

set -euo pipefail
CONV_FILE=/mnt/conv/testfile
ZNS_FILE=/mnt/zns/testfile
FILL_SIZE=1600m   # ~80% of 2GB
RUNTIME=60

# ── 1) Conventional F2FS — 초기 카운터 + fill ────────
printf '[1/4] NVMe F2FS 초기 fill (1,600 MB)...\n'
CONV_B=$(awk '{print $7}' /sys/block/nullb0/stat)

fio --name=conv_fill \
    --filename="${CONV_FILE}" \
    --size="${FILL_SIZE}" --bs=128k --rw=write \
    --ioengine=libaio --direct=1 --numjobs=1 \
    --output-format=json --output=/tmp/fio_conv_fill.json \
    2>/dev/null
printf '  ✔ fill 완료\n\n'

# ── 2) Conventional F2FS — 랜덤 쓰기 (GC 유발) ───────
printf '[2/4] NVMe F2FS 랜덤 쓰기 (%ds, GC 유발)...\n' "${RUNTIME}"
fio --name=conv_rand \
    --filename="${CONV_FILE}" \
    --size="${FILL_SIZE}" --bs=4k --rw=randwrite \
    --ioengine=libaio --direct=1 --numjobs=4 \
    --runtime="${RUNTIME}" --time_based \
    --output-format=json --output=/tmp/fio_conv_rand.json \
    2>/dev/null

sync
CONV_A=$(awk '{print $7}' /sys/block/nullb0/stat)
CONV_BLK=$(( (CONV_A - CONV_B) * 512 ))

CONV_APP=$(python3 << 'EOF'
import json
fill = json.load(open('/tmp/fio_conv_fill.json'))
rand = json.load(open('/tmp/fio_conv_rand.json'))
total = sum(j['write']['io_bytes'] for j in fill['jobs'] + rand['jobs'])
print(total)
EOF
)
echo "${CONV_APP} ${CONV_BLK}" > /tmp/conv_waf_stat
printf '  ✔ 앱: %d MB  |  블록: %d MB\n\n' \
    "$(( CONV_APP / 1024 / 1024 ))" "$(( CONV_BLK / 1024 / 1024 ))"

# ── 3) ZNS F2FS — 초기 카운터 + fill ────────────────
printf '[3/4] ZNS F2FS 초기 fill (1,600 MB)...\n'
ZNS_B=$(awk '{print $7}' /sys/block/nullb1/stat)

fio --name=zns_fill \
    --filename="${ZNS_FILE}" \
    --size="${FILL_SIZE}" --bs=128k --rw=write \
    --ioengine=libaio --direct=1 --numjobs=1 \
    --output-format=json --output=/tmp/fio_zns_fill.json \
    2>/dev/null
printf '  ✔ fill 완료\n\n'

# ── 4) ZNS F2FS — 랜덤 쓰기 (동일 워크로드) ─────────
printf '[4/4] ZNS F2FS 랜덤 쓰기 (%ds)...\n' "${RUNTIME}"
fio --name=zns_rand \
    --filename="${ZNS_FILE}" \
    --size="${FILL_SIZE}" --bs=4k --rw=randwrite \
    --ioengine=libaio --direct=1 --numjobs=4 \
    --runtime="${RUNTIME}" --time_based \
    --output-format=json --output=/tmp/fio_zns_rand.json \
    2>/dev/null

sync
ZNS_A=$(awk '{print $7}' /sys/block/nullb1/stat)
ZNS_BLK=$(( (ZNS_A - ZNS_B) * 512 ))

ZNS_APP=$(python3 << 'EOF'
import json
fill = json.load(open('/tmp/fio_zns_fill.json'))
rand = json.load(open('/tmp/fio_zns_rand.json'))
total = sum(j['write']['io_bytes'] for j in fill['jobs'] + rand['jobs'])
print(total)
EOF
)
echo "${ZNS_APP} ${ZNS_BLK}" > /tmp/zns_waf_stat
printf '  ✔ 앱: %d MB  |  블록: %d MB\n\n' \
    "$(( ZNS_APP / 1024 / 1024 ))" "$(( ZNS_BLK / 1024 / 1024 ))"

printf '완료 — 다음 단계: sudo bash calc_waf.sh\n'
bash — 호스트에서 파일 전송 
scp -P 2222 workload.sh user@localhost:~/
✅ chmod +x workload.sh && sudo bash workload.sh — 완료까지 약 3~5분 소요
⚠️ --direct=1은 페이지 캐시를 우회해 블록 장치에 직접 쓰며, 이후 sync로 남은 writeback을 플러시합니다. 이 두 가지를 함께 사용해야 write_sectors 스냅샷이 실제 기록량과 일치합니다.
Step 3.3 — WAF 계산 및 분석
1

WAF 계산 및 비교 출력 calc_waf.sh

저장된 스냅샷에서 앱 기록량(fio io_bytes)과 블록 장치 기록량(stat write_sectors)을 읽어 WAF를 계산하고 두 장치를 비교합니다.

calc_waf.sh 
#!/usr/bin/env bash
# calc_waf.sh — F2FS WAF 계산 및 비교 출력
# 사용법: sudo bash calc_waf.sh

set -euo pipefail

sep() { printf '%s\n' "$(printf '─%.0s' {1..58})"; }

if [[ ! -f /tmp/conv_waf_stat || ! -f /tmp/zns_waf_stat ]]; then
    printf '오류: workload.sh를 먼저 실행하세요.\n'; exit 1
fi

read CONV_APP CONV_BLK < /tmp/conv_waf_stat
read ZNS_APP  ZNS_BLK  < /tmp/zns_waf_stat

if [[ "${CONV_APP}" -eq 0 || "${ZNS_APP}" -eq 0 ]]; then
    printf '오류: 기록량이 0입니다. workload.sh를 다시 실행하세요.\n'; exit 1
fi

CONV_APP_MB=$(( CONV_APP / 1024 / 1024 ))
CONV_BLK_MB=$(( CONV_BLK / 1024 / 1024 ))
ZNS_APP_MB=$(( ZNS_APP   / 1024 / 1024 ))
ZNS_BLK_MB=$(( ZNS_BLK   / 1024 / 1024 ))

CONV_WAF=$(awk "BEGIN{printf \"%.2f\", ${CONV_BLK}/${CONV_APP}}")
ZNS_WAF=$(awk  "BEGIN{printf \"%.2f\", ${ZNS_BLK}/${ZNS_APP}}")

sep
printf '  📊  F2FS 파일시스템 레벨 WAF 비교\n'
sep
printf '  %-24s  %-10s  %-10s  %-6s\n' "Device" "App Write" "Blk Write" "WAF"
sep
printf '  %-24s  %-10s  %-10s  %-6s\n' \
    "NVMe F2FS (nullb0)" "${CONV_APP_MB} MB" "${CONV_BLK_MB} MB" "${CONV_WAF}"
printf '  %-24s  %-10s  %-10s  %-6s\n' \
    "ZNS  F2FS (nullb1)" "${ZNS_APP_MB} MB"  "${ZNS_BLK_MB} MB"  "${ZNS_WAF}"
sep
printf '\n  NVMe WAF > 1.0: F2FS 세그먼트 GC Copy-back으로 추가 블록 기록\n'
printf '  ZNS  WAF ≈ 1.0: Zone 단위 GC로 Copy-back 최소화\n\n'
bash — 호스트에서 파일 전송 
scp -P 2222 calc_waf.sh user@localhost:~/
✅ chmod +x calc_waf.sh && sudo bash calc_waf.sh

  Device                    App Write   Blk Write   WAF
──────────────────────────────────────────────────────────
  NVMe F2FS (nullb0)        2435 MB     3408 MB     1.40
  ZNS  F2FS (nullb1)        2441 MB     2563 MB     1.05
──────────────────────────────────────────────────────────

📌 null_blk은 메모리 기반 장치이므로 I/O 속도가 매우 빠릅니다. GC가 백그라운드에서 충분히 돌지 못하면 NVMe WAF가 예상보다 낮게 나올 수 있습니다. 이 경우 workload.sh의 RUNTIME 값을 120 이상으로 늘려 재측정하세요.
2

F2FS GC 동작 구조 비교

같은 랜덤 쓰기 워크로드를 받아도 NVMe F2FS와 ZNS F2FS의 GC 전략이 달라 WAF 차이가 발생합니다.

NVMe F2FS vs ZNS F2FS — GC Copy-back 구조 비교
NVMe F2FS (세그먼트 기반 GC)
──────────────────────────────────────────────────────────
  랜덤 4KB write  → F2FS: 새 세그먼트 끝에 append 기록
  이전 버전 페이지 → 해당 세그먼트에서 invalid 처리
  세그먼트 내 valid/invalid 혼재 시 GC 발생:
    [세그먼트 A: valid 10 + invalid 54 pages]
    → valid 10 pages를 새 세그먼트 B로 Copy-back   ← 추가 쓰기
    → 세그먼트 A Erase → 재사용
  블록 실제 기록량 > 앱 요청량  →  WAF > 1.0

ZNS F2FS (Zone 기반 GC)
──────────────────────────────────────────────────────────
  F2FS 세그먼트 경계 = Zone 경계 (64MB 정렬)
  랜덤 write → F2FS가 내부에서 순차 쓰기로 변환 후 Zone에 기록
  Zone 가득 참 → valid 비율 낮은 Zone 선정 (GC victim)
    → valid 페이지 수 적으면 Zone Reset만으로 재사용 가능   ← Copy-back 최소
  블록 실제 기록량 ≈ 앱 요청량  →  WAF ≈ 1.0
──────────────────────────────────────────────────────────
  핵심: Zone 크기(64MB) = F2FS GC 단위 → 부분 Copy-back 발생 구간 감소
✅ ZNS F2FS의 WAF 개선 핵심은 Zone Reset이 Erase 단위와 정확히 일치한다는 점입니다. Zone 전체가 invalid이면 복사 없이 Reset 하나로 재사용 가능하므로 GC Copy-back 비용이 대폭 줄어듭니다.

WAF 측정 결과표

장치 앱 기록량 (MB) 블록 기록량 (MB) WAF 비고
NVMe F2FS (/dev/nullb0) 기록하세요 기록하세요 기록하세요 세그먼트 GC, randwrite 60s
ZNS F2FS (/dev/nullb1) 기록하세요 기록하세요 기록하세요 Zone GC, randwrite 60s
📌 이 실습의 WAF는 파일시스템 레벨 측정값입니다. 실제 NAND 기반 SSD에서는 여기에 더해 펌웨어 레벨 GC WAF가 추가로 곱해져 최종 WAF는 더 높아집니다.
분석 질문

생각해보기

Q1. 이 실습의 WAF는 "파일시스템 레벨 WAF"입니다. null_blk에서 NAND 펌웨어 GC는 발생하지 않는데, F2FS WAF가 1보다 클 수 있는 이유는 무엇인가요?
힌트: F2FS의 로그 구조 쓰기 방식과 세그먼트 GC Copy-back이 블록 장치에 어떤 추가 I/O를 발생시키는지 생각해보세요.
Q2. ZNS F2FS에서 WAF가 낮은 이유를 Zone Reset 관점에서 설명하세요. Zone 내 valid 데이터가 아예 없는 경우와 일부 남아있는 경우 GC 동작은 어떻게 달라질까요?
힌트: Zone Reset은 Zone 전체를 한 번에 Erase합니다. Copy-back이 필요한 조건은 무엇인가요?
Q3. ZNS F2FS 포맷 시 zone_nr_conv=0이면 mkfs.f2fs -m이 실패합니다. F2FS가 ZNS 장치에서 Conventional Zone을 반드시 필요로 하는 이유는 무엇인가요?
힌트: F2FS의 슈퍼블록, 체크포인트, NAT 영역은 임의 위치에 덮어써야 합니다. Sequential Zone에서 이것이 가능할까요?
