실습 3: WAF(쓰기 증폭 지수) 비교 측정

Pre-conditioning · 혼합 워크로드 · WAF 공식 적용 및 ZNS vs Conventional 분석

실습 목표

WAF(Write Amplification Factor)는 호스트가 기록을 요청한 양 대비 실제로 낸드 플래시에 기록된 양의 비율입니다. Conventional SSD는 내부 가비지 컬렉션(GC) 때문에 WAF가 1보다 크게 높아지는 반면, ZNS SSD는 순차 쓰기만 허용되어 GC가 발생하지 않아 WAF가 1에 가깝습니다.

WAF = NAND 실제 기록량 / 호스트 요청 기록량
이상적인 값: WAF = 1.0  |  Conventional SSD (랜덤 쓰기 후 GC 발생 시): WAF ≫ 1.0

Step 3.1 — Pre-conditioning
1

통합 프리컨디셔닝 실행 precondition.sh

WAF 비교를 공정하게 하려면 전체 용량을 한 번 채워 GC가 발생할 수밖에 없는 환경을 만들어야 합니다. Conventional 장치는 랜덤 쓰기 2회 반복으로 Steady-State를 조성하고, ZNS는 순차 fill만 수행합니다.

⚙️
precondition.sh 동작 순서
  • Conventional(/dev/nvme0n1) 초기 write_sectors 기록
  • 전체 용량 순차 쓰기 1회 fill
  • 랜덤 쓰기 2회 반복 — GC Steady-State 조성
  • ZNS(/dev/nullb0) Zone Reset 후 순차 fill
  • 두 장치의 write_sectors 스냅샷 저장
precondition.sh 
#!/usr/bin/env bash
# precondition.sh — WAF 측정을 위한 사전 조건 구성
# 사용법: sudo bash precondition.sh

set -euo pipefail
CONV_DEV=/dev/nvme0n1
ZNS_DEV=/dev/nullb0
DEV_SIZE=16g   # 장치 전체 용량 (nvme_disk.raw 16GB에 맞춤)

# ── 1) Conventional 초기 카운터 ──────────────────────
printf '[1/5] Conventional 초기 카운터 기록...\n'
CONV_BEFORE=$(awk '{print $7}' /sys/block/nvme0n1/stat)
printf '      write_sectors = %s\n\n' "${CONV_BEFORE}"

# ── 2) Conventional 전체 순차 fill ───────────────────
printf '[2/5] Conventional 순차 fill...\n'
sudo fio \
  --name=precond_seq \
  --filename="${CONV_DEV}" \
  --ioengine=libaio --direct=1 \
  --rw=write --bs=128k --iodepth=8 \
  --size="${DEV_SIZE}" --numjobs=1 \
  --output-format=normal 2>/dev/null \
  | grep -oP '(?i)write:.*?BW=\K[^\s(]+' \
  | head -1 | xargs -I{} printf '  BW: {}\n' || true

# ── 3) Conventional 랜덤 쓰기 2회 (GC Steady-State) ──
printf '\n[3/5] Conventional 랜덤 쓰기 2회 (GC 유발)...\n'
for i in 1 2; do
    printf '  Pass %d...\n' "${i}"
    sudo fio \
      --name=precond_rand_"${i}" \
      --filename="${CONV_DEV}" \
      --ioengine=libaio --direct=1 \
      --rw=randwrite --bs=4k --iodepth=32 \
      --size="${DEV_SIZE}" --numjobs=1 \
      --output-format=normal 2>/dev/null \
      | grep -oP '(?i)write:.*?BW=\K[^\s(]+' \
      | head -1 | xargs -I{} printf '    BW: {}\n' || true
done

CONV_AFTER=$(awk '{print $7}' /sys/block/nvme0n1/stat)
echo "${CONV_BEFORE} ${CONV_AFTER}" > /tmp/conv_precond_stat
printf '  ✔ Conventional 프리컨디셔닝 완료\n\n'

# ── 4) ZNS 초기 카운터 + 순차 fill ──────────────────
printf '[4/5] ZNS Zone Reset 후 순차 fill...\n'
sudo blkzone reset "${ZNS_DEV}"
ZNS_BEFORE=$(awk '{print $7}' /sys/block/nullb0/stat)

sudo fio \
  --name=zns_precond \
  --filename="${ZNS_DEV}" \
  --ioengine=libaio --direct=1 --zonemode=zbd \
  --rw=write --bs=128k --iodepth=8 \
  --size="${DEV_SIZE}" --numjobs=1 \
  --output-format=normal 2>/dev/null \
  | grep -oP '(?i)write:.*?BW=\K[^\s(]+' \
  | head -1 | xargs -I{} printf '  ZNS BW: {}\n' || true

ZNS_AFTER=$(awk '{print $7}' /sys/block/nullb0/stat)
echo "${ZNS_BEFORE} ${ZNS_AFTER}" > /tmp/zns_precond_stat
printf '  ✔ ZNS 프리컨디셔닝 완료\n\n'

printf '[5/5] 완료 — 다음 단계: sudo bash workload.sh\n'
✅ chmod +x precondition.sh && sudo bash precondition.sh
⚠️ null_blk은 소프트웨어 에뮬레이터이므로 실제 GC는 발생하지 않습니다. 이 실습에서는 /sys/block/DEV/stat 기반 호스트 기록량을 비교하여 WAF ≈ 1인 기준값을 확인하는 데 초점을 맞춥니다.
Step 3.2 — 혼합 워크로드 인가
1

통합 워크로드 실행 workload.sh

Conventional 장치에는 랜덤 쓰기 70% + 순차 쓰기 30% 혼합 부하를, ZNS 장치에는 순차 쓰기를 60초 동안 인가하며 write_sectors를 모니터링합니다.

🗂️
workload.sh 동작 순서
  • Conventional 혼합 워크로드 60초 — write_sectors 전후 차 계산
  • ZNS 순차 워크로드 60초 — write_sectors 전후 차 계산
  • 두 장치 기록량 스냅샷을 /tmp/*_workload_stat에 저장
workload.sh 
#!/usr/bin/env bash
# workload.sh — Conventional + ZNS 워크로드 동시 측정
# 사용법: sudo bash workload.sh

set -euo pipefail
CONV_DEV=/dev/nvme0n1
ZNS_DEV=/dev/nullb0
RUNTIME=60
DEV_SIZE=16g   # precondition.sh와 동일한 크기로 비교

# ── 1) Conventional 혼합 워크로드 ────────────────────
printf '[1/2] Conventional 혼합 워크로드 (%ss)...\n' "${RUNTIME}"
CONV_B=$(awk '{print $7}' /sys/block/nvme0n1/stat)

sudo fio \
  --name=conv_mixed \
  --filename="${CONV_DEV}" \
  --ioengine=libaio --direct=1 \
  --rw=randrw --rwmixwrite=70 \
  --bs=4k --iodepth=16 \
  --size="${DEV_SIZE}" \
  --runtime="${RUNTIME}" --time_based --numjobs=1 \
  --output-format=normal 2>&1 | tee /tmp/fio_conv_mixed.log | grep -E 'IOPS|lat \(usec\)'

CONV_A=$(awk '{print $7}' /sys/block/nvme0n1/stat)
CONV_MB=$(( (CONV_A - CONV_B) * 512 / 1024 / 1024 ))
echo "${CONV_B} ${CONV_A} ${CONV_MB}" > /tmp/conv_workload_stat
printf '  ✔ Conventional 호스트 기록량: %s MB\n\n' "${CONV_MB}"

# ── 2) ZNS 순차 워크로드 ─────────────────────────────
printf '[2/2] ZNS 순차 워크로드 (%ss)...\n' "${RUNTIME}"
sudo blkzone reset "${ZNS_DEV}"
ZNS_B=$(awk '{print $7}' /sys/block/nullb0/stat)

sudo fio \
  --name=zns_seq \
  --filename="${ZNS_DEV}" \
  --ioengine=libaio --direct=1 --zonemode=zbd \
  --rw=write --bs=64k --iodepth=4 \
  --size="${DEV_SIZE}" \
  --runtime="${RUNTIME}" --time_based --numjobs=1 \
  --output-format=normal 2>&1 | tee /tmp/fio_zns_workload.log | grep -E 'IOPS|lat \(usec\)'

ZNS_A=$(awk '{print $7}' /sys/block/nullb0/stat)
ZNS_MB=$(( (ZNS_A - ZNS_B) * 512 / 1024 / 1024 ))
echo "${ZNS_B} ${ZNS_A} ${ZNS_MB}" > /tmp/zns_workload_stat
printf '  ✔ ZNS 호스트 기록량: %s MB\n\n' "${ZNS_MB}"

printf '완료 — 다음 단계: sudo bash calc_waf.sh\n'
✅ chmod +x workload.sh && sudo bash workload.sh
Step 3.3 — WAF 결과 분석
1

WAF 계산 및 비교 출력 calc_waf.sh

저장된 스냅샷 파일에서 호스트 기록량과 NAND 기록량을 읽어 WAF를 계산하고, ZNS와 Conventional의 구조적 차이를 분석합니다.

calc_waf.sh 
#!/usr/bin/env bash
# calc_waf.sh — WAF 계산 및 비교 출력
# 사용법: sudo bash calc_waf.sh

set -euo pipefail

sep() { printf '%s\n' "$(printf '─%.0s' {1..54})"; }

if [[ ! -f /tmp/conv_workload_stat || ! -f /tmp/zns_workload_stat ]]; then
    printf '오류: workload.sh를 먼저 실행하세요.\n'; exit 1
fi

read CONV_B CONV_A CONV_HOST_MB  < /tmp/conv_workload_stat
read ZNS_B  ZNS_A  ZNS_HOST_MB   < /tmp/zns_workload_stat

if [[ "${CONV_HOST_MB}" -eq 0 || "${ZNS_HOST_MB}" -eq 0 ]]; then
    printf '오류: 기록량이 0입니다. workload.sh를 다시 실행하세요.\n'; exit 1
fi

# null_blk: NAND 기록량 = 호스트 기록량 (실 GC 미발생)
CONV_NAND_MB=${CONV_HOST_MB}
ZNS_NAND_MB=${ZNS_HOST_MB}

CONV_WAF=$(awk "BEGIN{printf \"%.2f\", ${CONV_NAND_MB}/${CONV_HOST_MB}}")
ZNS_WAF=$(awk  "BEGIN{printf \"%.2f\", ${ZNS_NAND_MB}/${ZNS_HOST_MB}}")

sep
printf '  📊  WAF 비교 결과\n'
sep
printf '  %-20s  %-14s  %-14s  %-6s\n' "Device" "Host Write" "NAND Write" "WAF"
sep
printf '  %-20s  %-14s  %-14s  %-6s\n' \
    "Conventional (nvme0n1)" "${CONV_HOST_MB} MB" "${CONV_NAND_MB} MB" "${CONV_WAF}"
printf '  %-20s  %-14s  %-14s  %-6s\n' \
    "ZNS (nullb0)"           "${ZNS_HOST_MB} MB"  "${ZNS_NAND_MB} MB"  "${ZNS_WAF}"
sep
printf '\n  ※ null_blk 에뮬레이터: WAF ≈ 1.0 (실 GC 미발생)\n'
printf '     실 SSD에서 Conventional WAF는 랜덤 쓰기 시 2~10 이상 가능\n\n'

# 실 NVMe SSD WAF 측정 참고
printf '  [참고] 실 NVMe SSD WAF 측정:\n'
printf '  sudo nvme smart-log /dev/nvme0n1 | grep -E "data_units_written|host_write"\n'
✅ chmod +x calc_waf.sh && sudo bash calc_waf.sh

2

ZNS WAF ≈ 1인 이유 분석

GC Copy-back 발생 여부가 WAF 차이의 핵심입니다.

ZNS vs Conventional — GC 발생 구조 비교
Conventional SSD (랜덤 쓰기 → GC Copy-back 발생)
──────────────────────────────────────────────────────
  호스트 Write 4KB   → [Block A: page 3에 기록]
  GC 발생            → [Block A 유효 데이터 → Block C 복사]  ← 추가 NAND 쓰기
                     → [Block A Erase]
  NAND 실제 기록량   > 호스트 요청량  →  WAF > 1.0

ZNS SSD (순차 쓰기 + Zone Reset)
──────────────────────────────────────────────────────
  호스트 Write 64KB  → [Zone 0: WP 위치에 순차 기록]
  Zone 가득 참       → [Zone Reset = Block Erase, 복사 없음]
  재사용             → [Zone 0 처음부터 다시 쓰기]
  NAND 실제 기록량   = 호스트 요청량  →  WAF ≈ 1.0
──────────────────────────────────────────────────────
  차이: GC Copy-back 유 (Conventional) vs 무 (ZNS)
✅ ZNS에서 WAF ≈ 1이 가능한 핵심 이유는 Zone Reset이 Erase Block과 정확히 매핑되기 때문입니다. 유효 데이터 이동(Copy-back) 없이 Zone 전체를 한 번에 지우고 재시작하므로 추가 NAND 쓰기가 발생하지 않습니다.

생각해보기

Q1. 프리컨디셔닝을 하지 않으면 WAF 비교가 공정하지 않은 이유는 무엇인가요?
힌트: 새 SSD는 모든 블록이 미사용 상태입니다. GC가 발생하려면 어떤 조건이 갖춰져야 할까요?
Q2. ZNS SSD에서 WAF가 1에 가까워지면 SSD 수명(TBW: Total Bytes Written)에 어떤 영향을 줄까요?
힌트: NAND 플래시는 P/E Cycle 한계가 있습니다. WAF가 낮을수록 동일한 호스트 데이터 기록 시 NAND 소모가 어떻게 달라질까요?
Q3. ZNS 기반 시스템에서 WAF가 1보다 높아질 수 있는 시나리오를 생각해 보세요.
힌트: Zone 내에도 부분적으로 유효·무효 데이터가 섞이는 경우가 있을까요? 파일 시스템이 Zone 관리를 잘못하면 어떻게 될까요?
