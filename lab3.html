실습 3: F2FS 파일시스템 레벨 WAF 측정

Conv 환경 구성 · Conv WAF 측정 · ZNS 환경 구성 · ZNS WAF 측정 · 비교 분석

실습 목표

F2FS(Flash-Friendly File System)는 로그 구조 파일시스템으로, 파일시스템 레벨에서 자체 GC(Garbage Collection)를 수행합니다. 이 GC는 NAND 펌웨어 GC와 달리 null_blk 에뮬레이터에서도 실제로 동작하므로, 파일시스템 레벨 WAF가 1보다 클 수 있습니다.

Conventional SSD(/dev/nullb0)에 F2FS를 마운트해 랜덤 쓰기 부하를 주면, F2FS GC가 유효 데이터를 세그먼트 간에 복사(Copy-back)하여 블록 장치 실제 기록량이 앱 요청량보다 커집니다. ZNS SSD(/dev/nullb1)에서는 F2FS가 Zone 단위로 GC를 수행하므로 Copy-back이 줄어들어 WAF가 낮게 측정됩니다.

장치 초기화 안내 memory_backed=1 null_blk 장치는 데이터를 커널 메모리에 유지합니다. QEMU 메모리가 4GB로 제한되어 있어 nullb0(2GB)와 nullb1(2GB)을 동시에 올리면 메모리가 부족하므로 Conv와 ZNS를 순차적으로 측정합니다.
Step 3.1 — Conv F2FS 환경 준비

재부팅으로 기존 디스크 초기화 후 nullb0(2GB) 생성 및 F2FS 포맷/마운트 setup_conv.sh

lab1·2에서 사용하던 null_blk 장치와 backing memory를 완전히 초기화하기 위해 먼저 재부팅합니다. 재부팅 후 커널 모듈을 로드하고 nullb0을 2GB로 새로 생성합니다. nullb1은 이 단계에서 건드리지 않습니다.

bash — QEMU VM 내부 (재부팅) 
# 특정 커널로 재부팅하여 기존 null_blk 장치 초기화
sudo grub-reboot "Advanced options for Ubuntu>Ubuntu, with Linux 5.18.0-dpas"
sudo reboot
bash — QEMU VM 내부 (재부팅 후 — crc32 모듈 설치) 
# F2FS가 의존하는 crc32_generic 커널 모듈 빌드 및 설치
cd linux-5.18
make crypto/crc32_generic.ko
sudo cp crypto/crc32_generic.ko /lib/modules/$(uname -r)/kernel/crypto/
sudo depmod -a
sudo modprobe crc32_generic
⚙️
setup_conv.sh 동작 순서
  • null_blk 모듈 로드: modprobe null_blk nr_devices=0
  • f2fs 모듈 로드: modprobe f2fs
  • nullb0 (nvme0 configfs) 신규 생성, size=2048 (2GB)
  • f2fs-tools 설치 여부 확인 (mkfs.f2fs -V)
  • 마운트 포인트 /mnt/conv 생성
  • nullb0 포맷: mkfs.f2fs -f /dev/nullb0
  • /mnt/conv 마운트 후 df 확인
setup_conv.sh 
#!/usr/bin/env bash
# setup_conv.sh — create nullb0 (2 GB) after reboot and set up F2FS
# usage: sudo bash setup_conv.sh

set -euo pipefail

printf '── [1/5] Load kernel modules ────────────────────────\n'
modprobe null_blk nr_devices=0
modprobe f2fs

printf '── [2/5] Create nullb0 (2 GB) ───────────────────────\n'
if [ -d /sys/kernel/config/nullb/nvme0 ]; then
    echo 0 > /sys/kernel/config/nullb/nvme0/power
    rmdir /sys/kernel/config/nullb/nvme0
fi
mkdir /sys/kernel/config/nullb/nvme0
echo 4096 > /sys/kernel/config/nullb/nvme0/blocksize
echo 2048 > /sys/kernel/config/nullb/nvme0/size
echo 1    > /sys/kernel/config/nullb/nvme0/memory_backed
echo 1    > /sys/kernel/config/nullb/nvme0/power

printf '── [3/5] Check f2fs-tools ───────────────────────────\n'
if ! command -v mkfs.f2fs &>/dev/null; then
    apt-get install -y f2fs-tools
fi
mkfs.f2fs -V 2>&1 | head -1

printf '── [4/5] Create mount point ─────────────────────────\n'
mkdir -p /mnt/conv

printf '── [5/5] Format and mount F2FS ──────────────────────\n'
mkfs.f2fs -f /dev/nullb0
mount /dev/nullb0 /mnt/conv
df -h /mnt/conv
bash — 호스트에서 파일 전송 
scp -P 2222 setup_conv.sh user@localhost:~/
bash — QEMU VM 내부 
chmod +x setup_conv.sh && sudo bash setup_conv.sh

Step 3.2 — Conv F2FS WAF 측정

GC 유발 워크로드 실행 및 언마운트 workload_conv.sh

F2FS GC가 동작하려면 파일시스템이 충분히 채워진 상태에서 덮어쓰기(overwrite)가 일어나야 합니다. 먼저 1,600MB 파일을 순차 write로 채운 뒤, 같은 파일에 랜덤 4KB write를 60초 인가합니다. 덮어쓰인 페이지는 F2FS 세그먼트 내에서 무효(invalid) 상태로 남아 GC Copy-back을 유발합니다.

측정 완료 후 umount로 파일시스템을 언마운트하고 재부팅하여 backing memory를 완전히 초기화합니다. memory_backed=1 null_blk은 커널 내부 radix tree에 데이터를 유지하므로, 다음 단계(ZNS 2GB)를 위한 메모리 확보는 재부팅으로 처리합니다.

🗂️
workload_conv.sh 동작 순서
  • mkfs.f2fs -f + mount -o mode=lfs — LFS 모드로 재포맷·마운트 (WAF 극대화)
  • 1,600MB 파일 순차 fill — GC 유발 환경 조성
  • fill 완료 후 stat 기준점 기록 (randwrite 구간만 WAF 측정)
  • 동일 파일에 randwrite 60초 — F2FS invalid 페이지 누적
  • sync 후 write_sectors 최종값 기록 → ./conv_waf_stat 저장
  • umount /mnt/conv 후 재부팅 — backing memory 완전 초기화
workload_conv.sh 
#!/usr/bin/env bash
# workload_conv.sh — measure Conv F2FS WAF
# usage: sudo bash workload_conv.sh

set -euo pipefail
CONV_FILE=/mnt/conv/testfile
FILL_SIZE=1600m   # ~80% of 2GB
RUNTIME=60

printf '── [1/4] Format & Mount (LFS mode) ─────────────────\n'
umount /mnt/conv 2>/dev/null || true
mkfs.f2fs -f /dev/nullb0
mount -t f2fs -o mode=lfs /dev/nullb0 /mnt/conv

printf '── [2/4] Initial fill (1,600 MB) ────────────────────\n'
fio --name=conv_fill \
    --filename="${CONV_FILE}" \
    --size="${FILL_SIZE}" --bs=128k --rw=write \
    --ioengine=libaio --direct=1 --numjobs=1 \
    --output-format=json --output=/tmp/fio_conv_fill.json \
    2>/dev/null
sync

printf '── [3/4] Random write + WAF snapshot (%ds) ──────────\n' "${RUNTIME}"
CONV_B=$(awk '{print $7}' /sys/block/nullb0/stat)

fio --name=conv_rand \
    --filename="${CONV_FILE}" \
    --size="${FILL_SIZE}" --bs=4k --rw=randwrite \
    --ioengine=libaio --direct=1 --numjobs=4 \
    --runtime="${RUNTIME}" --time_based \
    --output-format=json --output=/tmp/fio_conv_rand.json \
    2>/dev/null

sync
CONV_A=$(awk '{print $7}' /sys/block/nullb0/stat)
CONV_BLK=$(( (CONV_A - CONV_B) * 512 ))

CONV_APP=$(python3 << 'EOF'
import json
rand = json.load(open('/tmp/fio_conv_rand.json'))
total = sum(j['write']['io_bytes'] for j in rand['jobs'])
print(total)
EOF
)
echo "${CONV_APP} ${CONV_BLK}" > ./conv_waf_stat
printf '  app: %d MB  |  blk: %d MB\n' \
    "$(( CONV_APP / 1024 / 1024 ))" "$(( CONV_BLK / 1024 / 1024 ))"

printf '── [4/4] Unmount ────────────────────────────────────\n'
umount /mnt/conv
bash — 호스트에서 파일 전송 
scp -P 2222 workload_conv.sh user@localhost:~/
bash — QEMU VM 내부 (완료까지 약 2~4분 소요) 
chmod +x workload_conv.sh && sudo bash workload_conv.sh
direct=1 + sync 병용 이유 --direct=1은 페이지 캐시를 우회해 블록 장치에 직접 쓰며, 이후 sync로 남은 writeback을 플러시합니다. 이 두 가지를 함께 사용해야 write_sectors 스냅샷이 실제 기록량과 일치합니다.
Step 3.3 — ZNS F2FS 환경 준비

재부팅 후 nullb1(2GB, zone_nr_conv=1) 생성 및 ZNS F2FS 포맷/마운트 setup_zns.sh

Conv 측정 완료 후 재부팅하여 backing memory를 초기화합니다. 재부팅 후 null_blk 모듈을 새로 로드하고 nullb1을 생성합니다. F2FS는 ZNS 장치에서 슈퍼블록, 체크포인트 등 메타데이터를 저장하기 위해 최소 1개의 Conventional Zone을 요구하므로, zone_nr_conv=1을 포함하여 생성합니다.

변경 전 (lab1/lab2) zone[ 0~31] = 32개 Sequential ← 모두 순차 쓰기 전용 변경 후 (lab3) zone[ 0 ] = 1개 Conventional ← F2FS 메타데이터 저장소 zone[ 1~31] = 31개 Sequential ← 데이터 영역 (총 용량 동일)
bash — QEMU VM 내부 (재부팅) 
# Conv 측정 완료 후 재부팅하여 backing memory 초기화
sudo grub-reboot "Advanced options for Ubuntu>Ubuntu, with Linux 5.18.0-dpas"
sudo reboot
⚙️
setup_zns.sh 동작 순서
  • null_blk 모듈 로드: modprobe null_blk nr_devices=0
  • f2fs 모듈 로드: modprobe f2fs
  • nullb1 (zns0 configfs) 신규 생성, zone_nr_conv=1 포함
  • 마운트 포인트 /mnt/zns 생성
  • nullb1 포맷: mkfs.f2fs -f -m /dev/nullb1 (ZNS 모드)
  • 마운트 후 blkzone report / df 확인
setup_zns.sh 
#!/usr/bin/env bash
# setup_zns.sh — create nullb1 (ZNS, zone_nr_conv=1) after reboot and set up ZNS F2FS
# usage: sudo bash setup_zns.sh

set -euo pipefail

printf '── [1/4] Load kernel modules ────────────────────────\n'
modprobe null_blk nr_devices=0
modprobe f2fs

printf '── [2/4] Create nullb1 (ZNS, zone_nr_conv=1) ────────\n'
if [ -d /sys/kernel/config/nullb/zns0 ]; then
    echo 0 > /sys/kernel/config/nullb/zns0/power
    rmdir /sys/kernel/config/nullb/zns0
fi
mkdir /sys/kernel/config/nullb/zns0
echo 2048 > /sys/kernel/config/nullb/zns0/size
echo 1    > /sys/kernel/config/nullb/zns0/zoned
echo 64   > /sys/kernel/config/nullb/zns0/zone_size
echo 1    > /sys/kernel/config/nullb/zns0/zone_nr_conv
echo 4096 > /sys/kernel/config/nullb/zns0/blocksize
echo 1    > /sys/kernel/config/nullb/zns0/memory_backed
echo 1    > /sys/kernel/config/nullb/zns0/power

printf '── [3/4] Create mount point ─────────────────────────\n'
mkdir -p /mnt/zns

printf '── [4/4] Format and mount ZNS F2FS ──────────────────\n'
mkfs.f2fs -f -m /dev/nullb1   # -m: ZNS mode (requires zone_nr_conv=1)
mount -t f2fs /dev/nullb1 /mnt/zns
df -h /mnt/zns
printf '\ndone  →  next: sudo bash workload_zns.sh\n'
bash — 호스트에서 파일 전송 
scp -P 2222 setup_zns.sh user@localhost:~/
bash — QEMU VM 내부 
chmod +x setup_zns.sh && sudo bash setup_zns.sh

Step 3.4 — ZNS F2FS WAF 측정

GC 유발 워크로드 실행 workload_zns.sh

Conv와 동일한 워크로드(1,600MB fill + randwrite 60초)를 nullb1에 적용하여 WAF 비교의 공정성을 유지합니다.

Step 3.5 — WAF 비교 분석

WAF 계산 및 비교 출력 calc_waf.sh

저장된 스냅샷에서 앱 기록량(fio io_bytes)과 블록 장치 기록량(stat write_sectors)을 읽어 WAF를 계산하고 두 장치를 비교합니다.

calc_waf.sh 
#!/usr/bin/env bash
# calc_waf.sh — compute and display F2FS WAF
# usage: sudo bash calc_waf.sh

set -euo pipefail

sep() { printf '%s\n' "$(printf '─%.0s' {1..58})"; }

if [[ ! -f ./conv_waf_stat || ! -f ./zns_waf_stat ]]; then
    printf 'error: run workload_conv.sh and workload_zns.sh first.\n'; exit 1
fi

read CONV_APP CONV_BLK < ./conv_waf_stat
read ZNS_APP  ZNS_BLK  < ./zns_waf_stat

if [[ "${CONV_APP}" -eq 0 || "${ZNS_APP}" -eq 0 ]]; then
    printf 'error: write count is 0. re-run workload scripts.\n'; exit 1
fi

CONV_APP_MB=$(( CONV_APP / 1024 / 1024 ))
CONV_BLK_MB=$(( CONV_BLK / 1024 / 1024 ))
ZNS_APP_MB=$(( ZNS_APP   / 1024 / 1024 ))
ZNS_BLK_MB=$(( ZNS_BLK   / 1024 / 1024 ))

CONV_WAF=$(awk "BEGIN{printf \"%.2f\", ${CONV_BLK}/${CONV_APP}}")
ZNS_WAF=$(awk  "BEGIN{printf \"%.2f\", ${ZNS_BLK}/${ZNS_APP}}")

sep
printf '  F2FS filesystem-level WAF comparison\n'
sep
printf '  %-24s  %-10s  %-10s  %-6s\n' "Device" "App Write" "Blk Write" "WAF"
sep
printf '  %-24s  %-10s  %-10s  %-6s\n' \
    "Conv F2FS (nullb0)" "${CONV_APP_MB} MB" "${CONV_BLK_MB} MB" "${CONV_WAF}"
printf '  %-24s  %-10s  %-10s  %-6s\n' \
    "ZNS  F2FS (nullb1)" "${ZNS_APP_MB} MB"  "${ZNS_BLK_MB} MB"  "${ZNS_WAF}"
sep
printf '\n  Conv WAF > 1.0: extra block writes due to F2FS segment GC copy-back\n'
printf '  ZNS  WAF ~= 1.0: minimal copy-back with zone-unit GC\n\n'
bash — 호스트에서 파일 전송 
scp -P 2222 calc_waf.sh user@localhost:~/
bash — QEMU VM 내부 
chmod +x calc_waf.sh && sudo bash calc_waf.sh

F2FS GC 동작 구조 비교

같은 랜덤 쓰기 워크로드를 받아도 Conv F2FS와 ZNS F2FS의 GC 전략이 달라 WAF 차이가 발생합니다.

Conv F2FS vs ZNS F2FS — GC copy-back structure
Conv F2FS (segment-based GC)
──────────────────────────────────────────────────────────
  random 4KB write  -> F2FS: appended to new segment tail
  stale pages        -> marked invalid in that segment
  GC triggered when segment has mixed valid/invalid pages:
    [segment A: valid 10 + invalid 54 pages]
    -> copy 10 valid pages to new segment B (copy-back)   ← extra write
    -> erase segment A  -> reuse
  actual block writes > app writes  ->  WAF > 1.0

ZNS F2FS (zone-based GC)
──────────────────────────────────────────────────────────
  F2FS segment boundary = zone boundary (64MB aligned)
  random write -> F2FS converts to sequential internally, writes to zone
  zone full -> select victim zone with low valid-page ratio (GC victim)
    -> if few valid pages, zone reset alone suffices for reuse   ← minimal copy-back
  actual block writes ~= app writes  ->  WAF ~= 1.0
──────────────────────────────────────────────────────────
  key: zone size (64MB) = F2FS GC unit -> fewer partial copy-back intervals
ZNS WAF 개선 핵심 ZNS F2FS의 WAF 개선 핵심은 Zone Reset이 Erase 단위와 정확히 일치한다는 점입니다. Zone 전체가 invalid이면 복사 없이 Reset 하나로 재사용 가능하므로 GC Copy-back 비용이 대폭 줄어듭니다.

생각해보기

Q1. 이 실습의 WAF는 "파일시스템 레벨 WAF"입니다. null_blk에서 NAND 펌웨어 GC는 발생하지 않는데, F2FS WAF가 1보다 클 수 있는 이유는 무엇인가요?
힌트: F2FS의 로그 구조 쓰기 방식과 세그먼트 GC Copy-back이 블록 장치에 어떤 추가 I/O를 발생시키는지 생각해보세요.
Q2. ZNS F2FS에서 WAF가 낮은 이유를 Zone Reset 관점에서 설명하세요. Zone 내 valid 데이터가 아예 없는 경우와 일부 남아있는 경우 GC 동작은 어떻게 달라질까요?
힌트: Zone Reset은 Zone 전체를 한 번에 Erase합니다. Copy-back이 필요한 조건은 무엇인가요?

힌트: F2FS의 슈퍼블록, 체크포인트, NAT 영역은 임의 위치에 덮어써야 합니다. Sequential Zone에서 이것이 가능할까요?
