Sequential Write 처리량 비교 · Random Write 위반 에러 체득

실습 목표

NVMe SSD(/dev/nullb0)와 ZNS 장치(/dev/nullb1)에 동일한 순차 쓰기를 수행하여 처리량(Throughput)을 비교합니다.

이어서 ZNS 장치에 임의 쓰기(Random Write)를 시도하여 Write Pointer 위반 에러가 발생하는 것을 직접 확인하고, ZNS의 쓰기 제약 조건을 체득합니다.

Step 1.1 — 순차 쓰기 처리량 비교
1

통합 벤치마크 실행 bench_seqwrite.sh

ZNS Zone 초기화 후 두 장치에 순차 쓰기를 수행하고 Bandwidth · Latency를 비교 출력합니다. /dev/nullb1은 zonemode=zbd로, /dev/nullb0은 일반 write로 실행됩니다.

🗂️
bench_seqwrite.sh 동작 순서
  • ZNS Zone 초기화 — blkzone reset /dev/nullb0
  • ZNS 순차 쓰기 — fio zonemode=zbd rw=write → /tmp/fio_zns_seq.log
  • NVMe SSD 순차 쓰기 — fio rw=write /dev/nullb0 → /tmp/fio_conv_seq.log
  • BW / Latency 자동 파싱 후 비교표 출력
bench_seqwrite.sh 
#!/usr/bin/env bash
# bench_seqwrite.sh — Sequential Write 처리량 비교 (ZNS vs Conventional)
# 사용법: sudo bash bench_seqwrite.sh

set -euo pipefail
ZNS_DEV=/dev/nullb1
CONV_DEV=/dev/nullb0
SIZE=2g
BS=64k
QD=4

sep() { printf '%s\n' "$(printf '─%.0s' {1..52})"; }

# ── 1) ZNS Zone 초기화 ───────────────────────────────
printf '[1/4] ZNS Zone 초기화 중...\n'
sudo blkzone reset "${ZNS_DEV}"
printf '      ✔ 완료 — 모든 Zone EMPTY\n\n'

# ── 2) ZNS 순차 쓰기 ─────────────────────────────────
printf '[2/4] ZNS 순차 쓰기 실행 (iodepth=%s)...\n' "${QD}"
sudo fio \
  --name=zns_seqwrite \
  --filename="${ZNS_DEV}" \
  --ioengine=libaio \
  --direct=1 \
  --zonemode=zbd \
  --rw=write \
  --bs="${BS}" \
  --iodepth="${QD}" \
  --size="${SIZE}" \
  --numjobs=1 \
  --group_reporting \
  --output-format=normal 2>&1 | tee /tmp/fio_zns_seq.log | grep -E 'IOPS|lat \(usec\)'

ZNS_BW=$(grep -oP '(?i)bw=\K[0-9]+(\.[0-9]+)?\s*(MiB|MB)/s' /tmp/fio_zns_seq.log | head -1 || echo 'N/A')
ZNS_LAT=$(grep -oP 'lat \(usec\):.*?avg=\K[0-9]+(\.[0-9]+)?' /tmp/fio_zns_seq.log | head -1 || echo 'N/A')
printf '      ✔ ZNS  BW=%s  Lat=%s µs\n\n' "${ZNS_BW}" "${ZNS_LAT}"

# ── 3) Conventional 순차 쓰기 ────────────────────────
printf '[3/4] Conventional 순차 쓰기 실행 (iodepth=%s)...\n' "${QD}"
sudo fio \
  --name=conv_seqwrite \
  --filename="${CONV_DEV}" \
  --ioengine=libaio \
  --direct=1 \
  --rw=write \
  --bs="${BS}" \
  --iodepth="${QD}" \
  --size="${SIZE}" \
  --numjobs=1 \
  --group_reporting \
  --output-format=normal 2>&1 | tee /tmp/fio_conv_seq.log | grep -E 'IOPS|lat \(usec\)'

CONV_BW=$(grep -oP '(?i)bw=\K[0-9]+(\.[0-9]+)?\s*(MiB|MB)/s' /tmp/fio_conv_seq.log | head -1 || echo 'N/A')
CONV_LAT=$(grep -oP 'lat \(usec\):.*?avg=\K[0-9]+(\.[0-9]+)?' /tmp/fio_conv_seq.log | head -1 || echo 'N/A')
printf '      ✔ CONV BW=%s  Lat=%s µs\n\n' "${CONV_BW}" "${CONV_LAT}"

# ── 4) 비교 출력 ─────────────────────────────────────
printf '[4/4] 결과 비교\n'
sep
printf '  %-16s  %-18s  %-14s\n' "Device" "Bandwidth" "Avg Latency"
sep
printf '  %-16s  %-18s  %-14s\n' "ZNS (nullb1)"   "${ZNS_BW}"  "${ZNS_LAT} µs"
printf '  %-16s  %-18s  %-14s\n' "NVMe (nullb0)" "${CONV_BW}" "${CONV_LAT} µs"
sep
printf '\n완료! 수치를 결과표에 기록하세요.\n'
bash — 호스트에서 파일 전송 
scp -P 2222 bench_seqwrite.sh user@localhost:~/
✅ chmod +x bench_seqwrite.sh && sudo bash bench_seqwrite.sh

Step 1.2 — Random Write 제약 확인
1

통합 벤치마크 실행 bench_randwrite.sh

ZNS 장치에 zonemode=none으로 Random Write를 시도하여 Write Pointer 위반 에러를 유발하고, Conventional 장치에서 동일 Random Write가 정상 동작함을 한 번에 비교합니다.

🚫 이 실험은 의도적으로 에러를 유발합니다. 에러 발생이 정상적인 결과입니다. 장치 손상은 없습니다.
🗂️
bench_randwrite.sh 동작 순서
  • ZNS Zone 초기화 후 zonemode=none randwrite 시도 → 에러 캡처
  • dmesg에서 Write Pointer 위반 로그 확인
  • Conventional 장치에 동일 Random Write → 정상 동작 확인
  • 두 결과 나란히 출력
bench_randwrite.sh 
#!/usr/bin/env bash
# bench_randwrite.sh — ZNS Random Write 제약 확인
# 사용법: sudo bash bench_randwrite.sh

set -euo pipefail
ZNS_DEV=/dev/nullb1
CONV_DEV=/dev/nullb0

sep() { printf '%s\n' "$(printf '─%.0s' {1..52})"; }

# ── 1) ZNS Random Write 시도 (에러 유발) ─────────────
printf '[1/3] ZNS Zone 초기화 후 Random Write 시도...\n'
sudo blkzone reset "${ZNS_DEV}"

sudo fio \
  --name=zns_randwrite \
  --filename="${ZNS_DEV}" \
  --ioengine=libaio \
  --direct=1 \
  --zonemode=none \
  --rw=randwrite \
  --bs=4k \
  --iodepth=1 \
  --size=64m \
  --numjobs=1 \
  --output-format=normal 2>&1 | tee /tmp/fio_zns_rand.log || true

ZNS_ERR=$(grep -oP 'error.*$|Invalid argument' /tmp/fio_zns_rand.log | head -1 || echo '에러 없음')
printf '      ZNS 결과: %s\n\n' "${ZNS_ERR}"

# ── 2) dmesg 위반 로그 확인 ──────────────────────────
printf '[2/3] 커널 에러 로그 확인...\n'
sudo dmesg | grep -i 'zone\|write pointer\|unaligned' | tail -5 || true
printf '\n'

# ── 3) Conventional Random Write (정상) ──────────────
printf '[3/3] Conventional Random Write 실행...\n'
sudo fio \
  --name=conv_randwrite \
  --filename="${CONV_DEV}" \
  --ioengine=libaio \
  --direct=1 \
  --rw=randwrite \
  --bs=4k \
  --iodepth=1 \
  --size=64m \
  --numjobs=1 \
  --output-format=normal 2>&1 | tee /tmp/fio_conv_rand.log | grep -E 'IOPS|lat \(usec\)'

CONV_BW=$(grep -oP '(?i)bw=\K[0-9]+(\.[0-9]+)?\s*(MiB|MB)/s' /tmp/fio_conv_rand.log | head -1 || echo 'N/A')

# ── 결과 요약 ─────────────────────────────────────────
sep
printf '  %-16s  %s\n' "Device" "Random Write 결과"
sep
printf '  %-16s  %s\n' "ZNS (nullb1)"   "❌ 에러: ${ZNS_ERR}"
printf '  %-16s  %s\n' "NVMe (nullb0)" "✔  정상: BW=${CONV_BW}"
sep

생각해보기

Q1. 순차 쓰기에서 ZNS와 Conventional SSD의 처리량 차이가 작은 이유는 무엇일까요?
힌트: null_blk은 메모리 기반 에뮬레이터입니다. 실제 낸드 플래시 기반 SSD에서는 GC 개입 여부가 차이를 만듭니다.
Q2. ZNS에서 Random Write를 허용하지 않는 이유는 무엇일까요? (낸드 플래시 물리 구조와 연관지어 설명하세요)
힌트: 낸드 플래시는 Erase Block 단위로만 초기화가 가능합니다. 임의 쓰기를 허용하면 내부적으로 어떤 작업이 필요할까요?
Q3. ZNS에서 기존 애플리케이션이 Random Write를 수행해야 한다면 어떤 소프트웨어 계층이 이를 처리해줄 수 있을까요?
힌트: 강의 슬라이드의 Linux Zoned Storage Support 다이어그램에서 dm-zoned와 F2FS의 역할을 확인하세요.
